# Session Journal — 2026-03-15

## Goal

Build the MCP server adapter (v0.2 high-priority item) — expose context-hooks' knowledge store, memo system, commit queries, and analysis tools to any MCP-speaking agent via a universal stdio server.

## What Was Done

### 1. Design & spec

Brainstormed the MCP adapter design through 4 clarifying questions:
- **Scope:** Both general-purpose MCP + agent-bridge drop-in replacement (compat mode)
- **Surface:** Full — all 14 agent-bridge tools + 9 context-hooks-unique tools = 23 total
- **Per-project:** One MCP server instance per project, resolved from cwd/git root
- **Zero-dep:** Thin protocol shim (~200 lines), no FastMCP

Wrote spec at `docs/superpowers/specs/2026-03-15-mcp-server-adapter-design.md`. Two review passes caught: Content-Length framing (not newline JSON), lazy thread_id generation, missing priority column on memos, maturity param on store_knowledge, and 4 other issues.

### 2. Implementation plan

11 tasks across 5 chunks, TDD throughout. Plan review caught 2 issues: missing tests for xref/profile handlers, private `_memo_to_dict` access.

### 3. Schema changes (Tasks 1-3)

- `lib/db.py`: Added `shared_state` table (key/value/updated_by/updated_at) with upsert/get/delete methods. Added `priority` column to memos. Updated `insert_memo()` to accept priority.
- `lib/knowledge.py`: Added optional `maturity` param to `store()` (default stays 'decision').

### 4. MCP protocol shim (Task 4)

- `lib/mcp.py`: 113-line `MCPServer` class implementing JSON-RPC 2.0 over stdin/stdout with Content-Length framing. Tool registry via `register_tool()`. Handles initialize, tools/list, tools/call, ping, notifications. Zero deps.

### 5. Tool handlers (Tasks 5-7)

- `lib/mcp_tools.py`: `build_handlers(ctx)` returns 23 handler closures over shared context. Each opens a fresh DB per call (avoids WAL lock contention).
  - 8 knowledge tools (store, search, get, list, promote, archive, restore, supersede)
  - 6 memo tools (send, check, read, reply with lazy threads, broadcast with priority, list_threads)
  - 3 task/state tools (handoff_task as structured memo, set/get shared_state)
  - 6 query/analysis tools (commits, parity, xref, health, profile, project_context)

### 6. Registration + CLI (Tasks 8-10)

- `TOOL_SCHEMAS`: 23 entries with JSON schemas + 14 agent-bridge compat aliases
- `register_all_tools(server, ctx, compat=None)`: Registers native + optional alias names
- `main()` with argparse: `--compat=agent-bridge`, `--project=/path`
- `bin/context-hooks mcp` dispatcher case + help text

### 7. Results

- 200 tests passing (up from 146). 9 implementation commits.
- `bin/context-hooks mcp --help` works
- 23 native tools, 37 with compat mode

## Session End (session 1)

## Session 2 — Dogfooding & quick wins

### Goal

Read and respond to KADE2's welcome memo, then fix the real issues surfaced by cross-agent memo exchange.

### 8. Cross-agent memo exchange

Read KADE2's welcome memo (#1 in context-hooks DB, #88 reply sent to KADE2 DB). Triaged the 4 tasks: 3 already done (tests, schema, priority column), 1 open (cross-project routing). Hit two real dogfooding issues during the exchange:

1. **Schema drift** — KADE2's DB was created by v0.1 code (no `priority` column). `ContextDB.__init__` runs `CREATE TABLE IF NOT EXISTS` which is a no-op on existing tables with different columns. Had to use raw SQL to send the reply.
2. **CLI syntax** — KADE agent tried `memo send --from X --subject Y` but the CLI only accepted positional args. Two garbled memos landed (from_agent="--from", subject="kade-agent").

Received two follow-up memos from KADE agent: design constraint (knowledge durability = dual-write) and bug report confirming both issues.

### 9. Quick Win 1: Schema migration (TDD)

- `lib/db.py` — Added `CURRENT_SCHEMA_VERSION`, `schema_version` table in SCHEMA, `_migrate()` method on `ContextDB.__init__`, `_migrate_v1_to_v2()` for adding priority column via `ALTER TABLE`. Version detection: no rows in schema_version + existing tables = v1; no rows + no tables = fresh (set to current).
- `tests/test_db.py` — 7 new tests: v1 gets priority column, gets shared_state table, data survives, insert with priority works post-migration, version tracked, idempotent, fresh DB starts at current version.

### 10. Quick Win 2: CLI flag syntax for memo send (TDD)

- `lib/knowledge.py` — New `parse_memo_send_args()` function. Detects flag vs positional syntax by checking if first arg starts with `--`. Supports `--from`, `--subject`, `--content`, `--to`, `--priority`, `--project`. Positional syntax (`<from> <subject> <content>`) still works. Updated `main()` to use parser + resolve `--project` via `data_dir()`.
- `lib/knowledge.py` — `send_memo()` now accepts `priority` kwarg.
- `tests/test_knowledge.py` — 8 new tests: flag syntax, --to, default broadcast, --priority, positional compat, --project parsed, --project sends to other DB, missing flags error.

### 11. Results

- 215 tests passing (up from 200). All existing tests unaffected.
- KADE2 DB can now be auto-migrated on open.
- Agents can send cross-project memos with `memo send --from X --subject Y --content Z --project /path/to/repo`.

## Session End

**Status:** Schema migration and CLI flag syntax complete. Cross-project memo exchange proven via dogfooding (KADE2 <-> context-hooks). Next: design cross-project memo routing layer, then tackle knowledge durability (dual-write per KADE2's design constraint memo).
